# /translate — ハーネスの心臓部

## 概要

ユーザーの指示を「ユーザー語 → エンジニア語 → リポジトリの具体箇所」に2段階翻訳する。
コードを修正する前に必ず実行し、翻訳結果をユーザーに確認させる。

---

## Step 0：論点の抽出

ユーザーの発言から**すべての指示を漏れなく**列挙する。

### ルール
- 省略・統合・要約は**しない**。ユーザーが3つ言ったら3つ出す
- 「ついでに」「あと」「それと」の後に続く指示も独立した論点として扱う
- 感情表現（怒り、皮肉、強調）は論点の優先度と重大度の手がかりとして扱う。**無視するな**
- 主語が省略されている場合は文脈から特定を試みる

### 出力形式
```
論点1: 「〜」
論点2: 「〜」
論点3: 「〜」
```

---

## Step 1：エンジニア語への翻訳

各論点を技術的な問題文に変換する。

### ルール
- 曖昧な表現を技術的に具体化する
- 「おかしい」→「〜にバグがある可能性」
- 「変えて」→「〜の値/ロジック/表示を変更する」
- 修正の種類を明示：バグ修正 / 機能追加 / 表示変更 / リファクタリング

### 出力形式
```
論点1: 〜 → [技術的問題文]
論点2: 〜 → [技術的問題文]
```

---

## Step 2：リポジトリの具体箇所への紐付け

各論点をリポジトリの具体的なファイル・関数・行に紐付ける。

### ルール
- ファイルパスと行番号まで特定する（例: `src/server.ts:42`）
- 複数箇所に影響する場合はすべて列挙する
- 不明な場合は「要調査」と明記し、調査方法を提示する

### 出力形式
```
論点1: [ファイルパス:行番号] [関数名/処理名]
論点2: [ファイルパス:行番号] [関数名/処理名]
```

---

## 翻訳完了後

翻訳結果をユーザーに提示し、確認を求める。

```
翻訳結果を確認してください。
「違う、そこじゃない」があれば、コードを触る前に今お知らせください。
```

ユーザーが確認・修正したら、初めてコードの修正に進む。
